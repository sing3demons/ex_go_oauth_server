# คัมภีร์เจาะลึก OIDC Discovery (OpenID Connect Discovery)

หากคุณกำลังสร้างระบบ Login หรือตัวกลางเชื่อมต่อ (Identity Provider) สิ่งที่ขาดไม่ได้คือ **OIDC Discovery (`/.well-known/openid-configuration`)** 
เอกสารนี้เปรียบเสมือน "คู่มือการคุยกับเซิร์ฟเวอร์" (Server Metadata) ที่นักพัฒนาแอปมือถือหรือเว็บ (Client) ทั่วโลกใช้เป็นมาตรฐานในการเซ็ตอัประบบให้เข้ากันแบบอัตโนมัติ

มาดูเบื้องลึกของการตั้งค่าเหล่านั้น ว่ามันออกแบบมาเพื่อประสานเนื้อหากันยังไง และมีประโยชน์ยังไงบ้าง

---

## 1. กลุ่ม "เส้นทางจราจร" (Endpoint Operations)
กลุ่ม URL เหล่านี้เป็นรากฐานของระบบ **OAuth 2.0 Flow** การระบุค่าเหล่านี้ จะบอก Client ว่าต้องเตะผู้ใช้ไปที่ไหน (Redirect) และขอข้อมูลได้ที่ช่องทางใด

*   **`authorization_endpoint`**
    *   **ทำอะไร**: เป็นประตูหน้าด่านที่เอาไว้โชว์ UI เช่น หน้า Login หรือจอ Consent (ขออนุญาตสิทธิ์)
    *   **ทำงานคู่กับ**: `response_types_supported` (มักจะตอบเป็น `code`)
*   **`token_endpoint`**
    *   **ทำอะไร**: ประตูหลังบ้าน! หลังจาก Client ได้ `code` มาแล้ว (ผ่าน URL) จะต้องแอบเรียก API มาที่นี่เพื่อรับ **Access Token** และ **ID Token** กลับไป
    *   **ทำงานคู่กับ**: `grant_types_supported` และกลไก PKCE
*   **`userinfo_endpoint`**
    *   **ทำอะไร**: ด่านตรวจสอบข้อมูลส่วนตัว (Profile/Email) โดยใช้ Access Token วางตรง HTTP Header แบบ Bearer
    *   **ทำงานคู่กับ**: `claims_supported` เพราะด่านนี้จะแจกเฉพาะข้อมูลที่เคยระบุใน Discovery ไว้เท่านั้น
*   **`end_session_endpoint`**
    *   **ทำอะไร**: ท่อระบายน้ำทิ้ง! ใช้ทำ **RP-Initiated Logout** เวลาผู้ใช้อยากออกจากระบบจากทุกอุปกรณ์แบบเด็ดขาด

---

## 2. กลุ่ม "ป้อมปราการและการลงนาม" (Signatures & Keys)
ในโลกของ OpenID ความน่าเชื่อถือคือที่หนึ่ง! ถ้าไม่มีกลุ่มนี้ ใครๆ ก็แฮก Token ตัวปลอมมายัดเยียดให้ระบบเราได้

*   **`issuer`**
    *   **ระดับความสำคัญ**: ⭐⭐⭐⭐⭐ (สูงสุด)
    *   **ทำอะไร**: URL ประจำตัวเซิร์ฟเวอร์เรา (เช่น `https://auth.mycompany.com`)
    *   **ความเชื่อมโยง**: ตอนผู้ใช้เข้าแอปด้วย **ID Token** (JWT) ภายใน Payload ของมันจะมีฟิลด์ `"iss"` อยู่.. ฝั่ง Client **บังคับ** ว่าต้องเอาค่า "iss" ใน Payload มาเทียบกับ "issuer" ในหน้า Discovery ว่าตรงกันเป๊ะ 100% ถ้าไม่ตรง = โยนทิ้ง!
*   **`jwks_uri`**
    *   **ทำอะไร**: URL ที่แจก **Public Keys (แม่กุญแจสาธารณะ)** 
    *   **ความเชื่อมโยง**: เราเป็นคนสร้าง Token ด้วย Private Key คราวนี้พอฝั่ง Client ถือ Token เราไว้ เขาอยากรู้ว่า "นี่ของแท้ไหม" เขาก็ไปดาวน์โหลดกุญแจที่ JWKS มาส่องดู ถ้าไขสำเร็จ แปลว่ารอยประทับตรานั้นเป็นของเราแน่นอน
*   **`id_token_signing_alg_values_supported`**
    *   **ทำอะไร**: คอยกระซิบบอก Client ว่า "ฉันใช้ดินสอสีอะไรเซ็นชื่อ" (เช่น `RS256`, `ES256`, หรือตัวเทพระดับประเทศอย่าง `EdDSA`) เพื่อให้เขาเตรียมฟังก์ชันถอดรหัสของภาษาตัวเองไว้ถูกตัว

## 3. กลุ่ม "ควบคุมพฤติกรรมและการยืนยันตัวตน" (Security Behaviors)
เราไม่เชื่อใจใครง่ายๆ! ดังนั้นเซิร์ฟเวอร์จะต้องประกาศ "กฎ" ว่าถ้าใครฝืนกฎนี้ เราไม่ให้ผ่าน

*   **`token_endpoint_auth_methods_supported`**
    *   **ทำอะไร**: กฎเหล็กตอนมาขอแลก Token 
    *   **ตัวอย่าง**:
        *   `client_secret_basic` : เอารหัส Client ไปทำ Base64 แล้วใส่บน Authorization Header (มาตรฐานหลัก)
        *   `client_secret_post` : แอบแปะรหัสลงมาในกล่อง Body (ส่วนใหญ่นิยมใช้เวลาระบบเก่ายิง Header ลำบาก)
        *   `none` : "ไม่ต้องมีรหัส" (ใช้สำหรับแอป Mobile App หรือ SPA ที่ซ่อนรหัสไม่ได้)
    *   **ความเชื่อมโยง**: ทำงานร่วมกับ `grant_types` ถ้าเลือก `none` แปลว่าคุณควรต้องบังคับให้ระบบใช้กลไกของ "กลุ่มถัดไป" (PKCE) ให้แน่นหนาที่สุดตายแทน!
*   **`code_challenge_methods_supported`**
    *   **ทำอะไร**: เป็นฟีเจอร์กันการโฉบขโมย Code ระหว่างทาง (Interception Attack) โดยเฉพาะสำหรับแอปมือถือ 
    *   **ความเชื่อมโยง**: เซิร์ฟเวอร์จะบอกว่ารองรับ `S256` (การใช้คณิตศาสตร์แฮชแบบวงกต SHA-256) ใครดักฟังรหัสก็จะเอาตัวแปลงกลับไม่ได้!

## 4. กลุ่ม "ขอบเขตข้อมูลและอัตลักษณ์" (Scopes & Identities)
กลุ่มสุดท้ายคือการรับรู้ว่าระบบนี้มีทรัพยากร (Resource) ให้อะไรแก่ลูกค้าได้บ้าง

*   **`scopes_supported`**
    *   **ทำอะไร**: แจกแจงว่าคุณขอข้อมูลอะไรผ่านหน้าต่าง Login ได้บ้าง
    *   **ตัวอย่าง**: `openid` (อันนี้บังคับต้องมี), `profile` (ได้หน้าตา-ชื่อ), `email` (ได้อีเมล)
*   **`claims_supported`**
    *   **ทำอะไร**: เมื่อขอ Scope ได้แล้ว.. แล้วของที่หยิบส่งให้ (Claim Parameter) มีหน้าตาไหนบ้าง? 
    *   **ตัวอย่าง**: ฉันมี `name`, `preferred_username`, `email_verified` แจกให้นะ
*   **`subject_types_supported`**
    *   **ทำอะไร**: บอกวิธีที่เซิร์ฟเวอร์เปิดเผยตัวเลขผู้ใช้งาน (`sub`) 
    *   **ความเชื่อมโยง**: 
        *   ถ้าใช้ `public` : ทุกแอปจะเห็นรหัสคุณตรงๆ (เช่น 12345) 
        *   ถ้าใช้ `pairwise` : เซิร์ฟเวอร์จะแอบแปลงรหัสคุณเป็นค่าสุ่มไม่ซ้ำกันตามแต่ Client! ทำให้ 2 แอปที่เป็นพาร์ทเนอร์กัน **"ไม่สามารถ"** เอาข้อมูลคุณมาเชื่อมโยงหรือตาม Track คุณได้ (ความปลอดภัยขั้นท็อปของ Privacy)

## 🎯 สรุป: ทำไมทุกค่าถึง "สอดคล้องกัน"
ระบบ OIDC ถูกสร้างมาให้ "กระจายศูนย์และอิสระ" (Decentralized Config).

แทนที่คุณจะต้องโทรไปสั่งสอนโปรแกรมเมอร์ที่ทำงานพาร์ทเนอร์คุณทีละคนว่าเชื่อมระบบคุณยังไง... **คุณแค่บอกว่า "เข้าไปดูไฟล์ `.well-known/openid-configuration` นะ"**

ระบบเฟรมเวิร์คฝั่งของพวกเขาอย่าง `NextAuth`, `Passport.js`, หรือ `Spring Boot` จะวิ่งมา **สแกนไฟล์ Discovery นี้แค่วินาทีเดียว**
ระบบฝั่งเขาจะปรับตัว (Auto Configure) ว่า:
1. "อ้าว! เจ้านี้ยืนยันความปลอดภัยด้วย `EdDSA` งั้นเปลี่ยน Algorithm แปป"
2. "เห้ย! ที่นี่บังคับใช้ PKCE แบบ `S256` ด้วย งั้นเดี๋ยวสร้าง Hash แปะเผื่อไปด้วยเลย"
3. "URL หน้า Login เค้าคืออันนี้นะ... โอเค โหลดกุญแจ Public Key จาก `jwks_uri` มาเก็บไว้รอเลย"

มันคือ **"ภาษากลางระดับโลก"** ที่ปิดประตูความผิดพลาดของ Programmer และทำให้อัตราโดนแฮกลดลงเกือบเป็น 0% ครับ!
